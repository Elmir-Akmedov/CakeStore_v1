The biggest bugs I see are these:

start_baking() trusts raw size input. It immediately uses recipe.get_price(size), recipe.get_bake_seconds(size), and SIZE_SLICES[size] with no validation, so an invalid size can crash the request with a KeyError. That is a real server-side bug, not just a polish issue. Fix it by validating size against SIZE_SLICES before any lookup and returning a clean error.
Your frontend sends POST requests as JSON with only Content-Type: application/json, but I do not see any CSRF token being attached in game.js. If your Django views are not explicitly csrf_exempt, most of these calls will fail. Fix it by adding CSRF handling in api() or by using a safe Django token flow.
_last_worker_tick is a module-global throttle. That works in one Python process, but it becomes unreliable if you run multiple server workers or restart the process, because each process has its own copy of that variable. The result can be duplicated or inconsistent worker actions. A DB-backed lock or a single authoritative tick job is safer.
Several model fields exist but are not really used by the simulation: service_speed, special_skill, and the multitasker / fresh_expert ideas are defined in Worker, but the engine mostly only uses bake_speed through bake_time_factor. That makes part of your game data decorative instead of functional.
The code mutates shared game state in many places without visible transaction protection: baking, fulfilling orders, hiring, firing, buying ovens, and ending the day all touch money, orders, and inventory. With polling and multiple requests, race conditions are possible. Wrap the critical actions in transaction.atomic() and lock the relevant rows when you update them.
The order timer in the UI is approximate, not exact. renderOrders() uses a fixed maxSec = 360, but real order patience ranges from 90 to 360 seconds and urgency changes at 60/30 seconds. That means the progress bar will look wrong for many orders. Use the actual expiry time or a per-order max duration from the backend.

Thought for 43s

This is the right moment to switch from “poll-driven simulation” to a server-authoritative game loop. Right now your frontend polls /api/state/ every 500ms, and that endpoint also advances the world by finishing cakes, running workers, generating orders, and expiring orders. That means a GET request is changing game state, so the client is effectively driving time. On top of that, all POST endpoints are csrf_exempt, and the views do very little input validation before handing values into the engine.

The loop I would move to

The backend should own time. The frontend should only:

send player commands,
fetch snapshots,
render state.

A cleaner loop looks like this:

def tick_game(now=None):
    now = now or timezone.now()
    with transaction.atomic():
        state = GameState.objects.select_for_update().get(pk=1)

        if not state.game_started:
            return
        if not state.is_open:
            return

        complete_finished_cakes(now)
        process_worker_actions(now)
        generate_due_orders(now)
        expire_due_orders(now)

        state.last_tick_at = now
        state.save(update_fields=['last_tick_at'])


Then:

GET /api/state/ becomes read-only snapshot data,
POST /api/tick/ is handled by a scheduler/background job, not the browser,
or, if you want the lightest possible setup, a management command runs the tick loop every second.

That is the biggest redesign you need.
